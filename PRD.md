## 1. 요약 (Executive Summary)

**꿈이든 / 내일탐색**은 **특수학교 특수교육대상자**를 위한 **직업 탐구 학습** 맥락에서, 학생이 AI 동반자 **이든(Eiden)** 과 대화하며 직업을 탐색하고, 담당 교사가 그 과정을 **수업 지원 콘솔**로 확인하는 웹 서비스다.

이 제품은 학생에게 **직업 적합성을 판정하지 않는다.** 대신 학생이 여러 직업을 보고, 궁금한 점을 표현하고, 다음 수업에서 이어갈 **관심·이해·지원 단서**를 남기는 것이 목표다.

현재 저장소는 React + Vite 기반 **프론트엔드 프로토타입(v0.1.0)** 이다. 학생 7단계 플로우, 교사 대시보드, localStorage 세션, mock 교사 로그가 구현되어 있으며, 백엔드·인증·실제 음성/AAC 연동은 아직 없다.

## 2. 배경과 문제

### 2.1 배경

- 특수학교에서 특수교육대상자의 **직업 탐구 학습**은 필수이지만, 기존 도구는 **점수·등급·적합률** 중심인 경우가 많다.
- 학생은 한 번에 완벽한 답을 내지 못할 수 있고, 이해가 어려울 때 **실패가 아니라 지원이 필요한 순간**이다.
- 교사는 수업 중 학생 반응을 놓치기 쉽고, 다음 수업 설계로 연결되는 기록 도구가 부족하다.

### 2.2 해결하려는 문제

| 이해관계자 | Pain Point |
|-----------|------------|
| 학생 | 직업 정보가 추상적이고, 정답 압박 없이 천천히 살펴볼 공간이 없음 |
| 교사 | 학생의 관심·어려움·지원 요청을 한곳에서 보고 다음 수업을 설계하기 어려움 |
| 학교/기관 | 평가형 직업검사와 달리 **특수학교·특수교육대상자 직업 탐구 학습에 맞는 대화형 탐색** 도구 부재 |

### 2.3 제품 포지셔닝

- **하지 않는 것:** 직업 매칭 점수, 적합/부적합 판정, 학생 순위, 실패/오답 프레이밍
- **하는 것:** 장면 단위 직업 체험, 관심 표현 수집, 지원 옵션 제공, 교사 확인 워크플로

## 3. 제품 비전과 목표

### 3.1 비전

> 학생이 표현한 관심과 어려움이 다음 수업 지원으로 자연스럽게 이어지는 직업 탐색 경험

### 3.2 제품 목표 (Goals)

1. **G1.** 학생이 3개 직업(바리스타, 도서관 사서, 제빵사)을 장면 단위로 탐색하고 자기 생각을 기록한다.
2. **G2.** 학생이 어려울 때 도움 요청·쉬기·그림 지원을 **문제 행동이 아닌 지원 옵션**으로 사용한다.
3. **G3.** 교사가 학생 반응을 `관심 표현`, `이해 단서`, `지원 필요`로 확인하고 지원 방식을 기록한다.
4. **G4.** 고등학생 대상에 맞게 **유아적이지 않고 차분한** UI/카피를 유지한다.

### 3.3 비목표 (Non-Goals) — v1

- 직업 적합도 알고리즘 / AI 추천 점수
- 학교 행정 시스템(출결, 성적) 연동
- 다수 학교 SaaS 멀티테넌시
- 완전한 음성 인식·TTS·AAC 하드웨어 연동 (프로토타입은 UI 시뮬레이션)

## 4. 사용자와 페르소나

### 4.1 Primary — 학생 (특수학교 특수교육대상자)

- **Needs:** 한 번에 하나의 질문, 시각 자료, 다시 듣기, 쉬어도 되는 흐름
- **Behavior:** 선택지 기반 반응, 짧은 문장 표현, 교사 도움을 부담 없이 요청
- **Success:** 「해보고 싶어요」「더 알아볼래요」 등 자기 표현을 남기고 다른 직업도 볼 수 있음

### 4.2 Secondary — 담당 교사

- **Needs:** 오늘 확인할 기록, 지원 필요 순간, 학생별 대화 흐름, 다음 수업 메모
- **Behavior:** drawer에서 빠르게 확인, 지원 방식 선택, 기록 완료 처리
- **Success:** 평가 없이 「다음 수업에서 그림 자료로 다시 보기」 같은 실행 가능한 지원 결정

### 4.3 Tertiary — PM/디자이너/개발 (내부)

- 디자인 잠금 문서·목업·E2E 스크린샷으로 피델리티 검수
- `design-qa.md` 기준 handoff 관리

## 5. 사용자 여정 (Core Journey)

```
랜딩 → 직업 길찾기 → 직업 소개 → 하루 체험 → 탐색 정리 → 저장 완료 → 내 기록
                                                              ↘
                                                    교사 대시보드 (병렬 진입)
```

### 5.1 학생 핵심 시나리오 (Happy Path)

1. 랜딩에서 서비스 소개를 읽고 **탐색 시작**
2. 직업 지도에서 **도서관 사서** 선택
3. 직업 소개에서 궁금한 활동(예: 책 정리) 선택
4. 하루 체험 4장면 중 질문에 답하며 진행 (필요 시 그림 지원·도움 요청)
5. 탐색 정리에서 기억 장면 + 생각(예: 더 알아볼래요) 선택 후 **기록 저장**
6. 저장 완료 확인 후 **내 기록**에서 다시 열람

### 5.2 교사 핵심 시나리오

1. 랜딩 또는 학생 화면 **교사용으로 보기** 진입
2. **확인 대기** 로그·지원 필요 순간 확인
3. 학생 행 클릭 → drawer에서 이든 요약·관찰 신호 확인
4. **이해 확인** / **그림 자료로 다시 보기** 등 지원 방식 선택
5. 상태 **기록 완료**로 갱신

### 7.5 품질 검증

```bash
npm ci
npm run verify   # build + frontend tests + server no-emit + PostgreSQL smoke + e2e + visual:diff
```

Release package manager: npm with `package-lock.json`. Do not use `pnpm-lock.yaml` for this release surface.

Required deployment environment:

```bash
DATABASE_URL=postgres://...
SESSION_SECRET=replace-with-random-secret
SERVER_ENCRYPTION_KEY=replace-with-random-secret
```

Deployment topology: single-origin Vite frontend + Express API + PostgreSQL runtime. The Vite build is served from the same public origin as `/api/*`, the Express API handles backend routes on that origin, and PostgreSQL stores runtime API state.

## 8. 디자인·UX 원칙

1. **평가 금지** — UI·카피·교사 콘솔 전역
2. **한 화면 한 질문** — 학생 인지 부하 최소화
3. **지원 = 옵션** — 도움/쉬기/AAC/그림은 실패가 아님
4. **학생 = 탐색 세계, 교사 = 지원 콘솔** — 화면 톤 분리
5. **고등학생 톤** — 유아용 게임 UI, 과도한 3D 장난감 미학 지양
6. **이든 = 참고, 교사 = 최종 확인**

상세 토큰: `DESIGN.md`, 잠금: `outputs/design-lock-summary.md`
